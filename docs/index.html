Verona Inferfaces Specification "Editor" 3.0.0

CC BY-SA 4.0

This is one part of the Verona Online Assessment Standards. All messages are sent via the postMessage function of the html page. The editor takes the page root of its parent as target (parent.window), and the application binds the function call to the iframe element of the editor. Most important, the message body carries as first parameter the operationId of the message. The html page MUST contain a <script>-tag with metadata. The syntax and structure of this data are described here.

Sub voeReadyNotification

Ready Notification

The editor announces that it's code is loaded and initialized so the communication can start.

Accepts the following message:

voeReadyNotification

Payload
object
metadata
required
string

Via this property, the editor sends the stringified metadata object definied as json-ld in the header of the html file. See here for more information.

Additional properties are allowed.

Pub voeStartCommand

Start Command

The application sends parameters for editing the unit and commands the start of user interaction.

Accepts the following message:

voeStartCommand

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

unitDefinition
string
byte

The definition of the unit (if given) to be edited.

unitDefinitionType
string

This lets the editor the unit definition format know. This might avoid UI mess after getting old definitions.

editorConfig
object

This data supplies some information or instruction about this specific editing of the unit.

definitionReportPolicy
string

The host expects the editor to send all changes when happening or only on demand. The latter might lead to better performance. In this case (default), the host triggers report by sending 'voeGetDefinitionRequest'.

Enum: "eager" "on-demand"

Additional properties are allowed.

Additional properties are allowed.

Examples

voeStartCommand
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk",
  "unitDefinition": "iqb-scripted::1.4.0 title::Testscript Title2??Hilfetext2 multiple-choice::mc_var1::1::Multiple Choice Feld: ::Choice1##Choice2##Choice3??Hilfetext1 drop-down::dd_var1::1::Dropdown Feld: ::Choice1##Choice2##Choice3??Hilfetext1 checkbox::check_var1::0::Ja klick mal! if-start::dd_var1::1 input-text::jajaj::sap osjxapsoxa if-else input-text::jajaj22::sap osjxapsoxa UUUU text::NOT Choice1 if-end",
  "unitDefinitionType": "iqb-scripted@1.4.0",
  "editorConfig": {
    "definitionReportPolicy": "eager"
  }
}

Sub voeDefinitionChangedNotification

Definition Changed Notification

The defintion of the unit has changed.

Accepts the following message:

voeDefinitionChangedNotification

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

timeStamp
required
string
date-time

Ensures, that later arriving states are ignored.

unitDefinition
string
byte

To be stored.

unitDefinitionType
string

In order to understand the defintion outside the editor, the type of the definition should be known. This helps to select a suitable player to run the unit.

variables
array<object>

Variable data help to prepare the processing of answers (coding scheme).

id
required
string
type
required
string

Data type of the value.

Enum: "string" "integer" "number" "boolean"
format
string

Some more information to specify the data type of the value. Every editor might introduce special formats to support answer processing. One example at IQB is 'marking' to store text markings in the form of '85-113-orange'.

multiple
boolean

If true, the value can contain multiple values as array.

values
array<object>

This list contains of possible values of the variable.

value
required
string
label
string

Label to describe the value. If - for example - there are options to select and these options are stored as numbers 1/2/3 etc., the label can help to identify the selected option in dialogs and analyses.

Additional items are allowed.

valuesComplete
boolean

If true, the list of possible values contains of ALL possible values.

Additional items are allowed.

Additional properties are allowed.

Examples

voeDefinitionChangedNotification
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk",
  "timeStamp": "2019-08-24T14:15:22Z",
  "unitDefinition": "string",
  "unitDefinitionType": "iqb-scripted@1.4.0",
  "variables": [
    {
      "id": "ME3491a",
      "type": "boolean",
      "format": "marking",
      "multiple": true,
      "values": [
        {
          "value": "1",
          "label": "I love Berlin."
        }
      ],
      "valuesComplete": true
    }
  ]
}

Pub voeGetDefinitionRequest

Get Definition Request

The application wants the editor to report the last state of the definition. The editor will answer by sending 'voeDefinitionChangedNotification' with full payload.

Accepts the following message:

voeGetDefinitionRequest

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

Additional properties are allowed.

Examples

voeGetDefinitionRequest
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk"
}
