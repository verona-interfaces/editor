Verona Inferfaces Specification "Editor" 2.0.0

CC BY-SA 4.0

This is one part of the Verona Online Assessment Standards. All messages are sent via the postMessage function of the html page. The editor takes the page root of its parent as target (parent.window), and the application binds the function call to the iframe element of the editor. Most important, the message body carries as first parameter the operationId of the message.

Sub voeReadyNotification

Ready Notification

The editor announces that it's code is loaded and initialized so the communication can start.

Accepts the following message:

voeReadyNotification

Payload
object
apiVersion
required
string
must match ^\d+\.\d+.\d+$

This lets the application know what verona API version the editor supports. The host will decide whether this declaration matches the requirements of the host or not and which features are not or differently implemented by the editor.

notSupportedApiFeatures
string

Space separated list of keys of features not implemented by the player. For details see corresponding player metadata doc!

supportedUnitDefinitionTypes
string

Space separated list of keys of unit definition formats supported by the player. After a '@' char, a version might be added (semver-format, caret and tilde might be used).

Additional properties are allowed.

Examples

voeReadyNotification
Payload 
{
  "apiVersion": "1.1.0",
  "notSupportedApiFeatures": "report-eager",
  "supportedUnitDefinitionTypes": "iqb-scripted@2.4.1"
}

Pub voeStartCommand

Start Command

The application sends parameters for editing the unit and commands the start of user interaction.

Accepts the following message:

voeStartCommand

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

unitDefinition
string
byte

The definition of the unit (if given) to be edited.

unitDefinitionType
string

This lets the editor the unit definition format know. This might avoid UI mess after getting old definitions.

editorConfig
object

This data supplies some information or instruction about this specific editing of the unit.

definitionReportPolicy
string

The host expects the editor to send all changes when happening or only on demand. The latter might lead to better performance. In this case (default), the host triggers report by sending 'voeGetDefinitionRequest'.

Enum: "eager" "on-demand"

Additional properties are allowed.

Additional properties are allowed.

Examples

voeStartCommand
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk",
  "unitDefinition": "iqb-scripted::1.4.0 title::Testscript Title2??Hilfetext2 multiple-choice::mc_var1::1::Multiple Choice Feld: ::Choice1##Choice2##Choice3??Hilfetext1 drop-down::dd_var1::1::Dropdown Feld: ::Choice1##Choice2##Choice3??Hilfetext1 checkbox::check_var1::0::Ja klick mal! if-start::dd_var1::1 input-text::jajaj::sap osjxapsoxa if-else input-text::jajaj22::sap osjxapsoxa UUUU text::NOT Choice1 if-end",
  "unitDefinitionType": "iqb-scripted@1.4.0",
  "editorConfig": {
    "definitionReportPolicy": "eager"
  }
}

Sub voeDefinitionChangedNotification

Definition Changed Notification

The defintion of the unit has changed.

Accepts the following message:

voeDefinitionChangedNotification

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

timeStamp
required
string
date-time

Ensures, that later arriving states are ignored.

unitDefinition
string
byte

To be stored.

Additional properties are allowed.

Examples

voeDefinitionChangedNotification
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk",
  "timeStamp": "2019-08-24T14:15:22Z",
  "unitDefinition": "string"
}

Pub voeGetDefinitionRequest

Get Definition Request

The application wants the editor to report the last state of the definition. The editor will answer by sending 'voeDefinitionChangedNotification' with full payload.

Accepts the following message:

voeGetDefinitionRequest

Payload
object
sessionId SessionIdString
required
string

The session id flags all communication. If a message has no or empty session id, it's not processed. The session id is unique and was generated by the application. Any simple algorithm would work. The session id helps to link the correct unit with the data of the message. Using the id of the editor hosting html element is less reliable, because the element could be reused with another unit.

Additional properties are allowed.

Examples

voeGetDefinitionRequest
Payload 
{
  "sessionId": "idk8ur5jf9ru5jk"
}
